# HearWords 完整需求文档

本文档基于当前项目实现与 [产品开发任务清单](D:\Projects\HearWords_Web\产品开发任务清单.md) 整理，目标是统一产品目标、功能范围、实现边界与阶段性交付标准，作为后续开发、测试、验收的主文档。

---

## 1. 产品定位

HearWords 是一个帮助英语学习者将图片、文档或文本中的陌生词汇，快速转化为“可筛选、可听写、可复习”的个人词库工具。

核心价值是把“被动阅读”转化为“主动记忆”，形成完整学习闭环：

`输入内容 -> 提取文本 -> 解析单词 -> 整理词库 -> 听写复习 -> 更新记忆状态`

---

## 2. 产品目标

### 2.1 核心目标

- 支持从多种内容来源中提取英文单词。
- 支持将候选单词整理为个人词库。
- 支持通过听写与复习机制巩固记忆。
- 支持本地优先运行，保证基础可用性。

### 2.2 当前阶段目标

当前项目更适合作为“单机版 Web MVP”持续迭代，重点是完善：

- 解析准确性
- 多格式导入能力
- 听写与复习体验
- 本地数据稳定性

不以多用户平台化能力为当前第一优先级，但需为未来扩展预留结构。

---

## 3. 用户与使用场景

### 3.1 目标用户

- 考研、雅思、托福等考试备考用户
- 日常英文阅读用户
- 需要从截图、课件、文章中整理单词的学习者
- 希望通过听写巩固拼写与记忆的用户

### 3.2 核心使用场景

- 用户上传一张英文截图，系统 OCR 后提取候选单词。
- 用户上传 PDF / Word / Excel / 文本文件，系统解析并生成候选单词。
- 用户勾选需要保留的单词，导入自己的单词书。
- 用户开始一轮待复习或全量听写任务。
- 用户通过听写结果更新熟练度，系统自动安排下次复习时间。

## 5. 功能需求

## 5.1 基础架构与通用模块

### FR-001 本地应用运行

系统应以本地 Node 服务 + 浏览器前端的形式运行。

验收要求：

- 可通过 `npm start` 启动。
- 浏览器打开本地地址后可以正常使用。

### FR-002 本地数据持久化

系统应将用户词库、解析历史、会话状态、设置持久化保存到本地。

验收要求：

- 重启服务后数据不丢失。
- 词库与听写会话可恢复。

### FR-003 多格式导入

系统应支持如下输入来源：

- 图片：`jpg`、`jpeg`、`png`
- 文本：`txt`、`md`、`csv`、`json`
- 文档：`pdf`、`docx`、`xlsx`

当前阶段说明：

- 图片 OCR 为正式功能。
- `pdf/docx/xlsx` 当前为轻量读取能力，后续应升级为真实解析能力。

---

## 5.2 解析引擎模块

### FR-101 OCR 流水线

系统应支持对图片进行 OCR 提取英文文本。

要求：

- 支持基础预处理。
- 支持自动旋转与纠偏。
- 支持去噪与增强。
- 输出识别文字与基础识别信息。

### FR-102 普通解析模式

系统应支持纯本地解析模式。

解析流程：

1. 文本清洗
2. 正则分词
3. 过滤标点、数字、无效词
4. 简单词形还原
5. 查询本地词典
6. 生成候选单词

### FR-103 增强解析模式

系统应支持 One-API 增强解析模式。

要求：

- 前端可切换增强模式。
- 服务端可配置 Base URL、Model、API Key、Timeout、System Prompt。
- 服务端要求 LLM 尽量返回结构化 JSON。
- 解析失败、超时或返回无效 JSON 时，必须自动降级为普通解析。

### FR-104 单词清洗与词形归一

系统应在解析时对单词进行清洗和归一化处理。

要求：

- 过滤数字
- 过滤无意义符号
- 过滤常见词性标记噪声
- 对复数、过去式、进行时等做基础词形还原
- 同一 lemma 在候选集中去重

### FR-105 候选词生成

每次解析应生成候选单词列表。

候选项至少应包含：

- `lemma`
- `rawWord`
- `definition`
- `pos`
- `exampleSentence`
- `sourceName`
- `bookName`

---

## 5.3 词库管理模块

### FR-201 导入候选词

用户可将候选词导入个人词库。

要求：

- 支持仅导入勾选项。
- 支持为导入项打上词书名和来源。
- 重复 lemma 应合并而不是生成重复词条。

### FR-202 单词书管理

系统应支持查看、搜索、筛选和维护词库。

要求：

- 按关键词搜索
- 按状态筛选：全部、待复习、新词、已熟练
- 按词书/来源筛选
- 支持编辑定义与例句
- 支持删除词条
- 支持导出词库

### FR-203 单词状态存储

词库记录至少应包含：

- 基础字段：`id`、`lemma`、`rawWord`
- 内容字段：`definition`、`pos`、`exampleSentence`、`phonetic`
- 来源字段：`sourceName`、`bookName`、`lastSource`、`originalForms`
- 学习状态：`masteryLevel`、`failCount`、`nextReviewTime`
- 时间字段：`createdAt`、`updatedAt`

---

## 5.4 听写与复习模块

### FR-301 听写轮次启动

系统应支持启动两类听写：

- 待复习单词
- 全量单词

### FR-302 听写展示

听写界面应至少展示：

- 当前进度
- 当前单词定义
- 当前例句
- 输入框
- 播放单词
- 播放例句
- 检查拼写
- 跳过
- 重置本轮

### FR-303 拼写判定

系统应支持用户输入拼写并判定正误。

要求：

- 大小写不敏感
- 正确时进入下一词
- 错误时返回差异提示和正确答案

### FR-304 复习排期

系统应根据答对、答错、跳过结果更新词条复习状态。

要求：

- 答对后提升熟练度，并更新下次复习时间
- 答错后增加失败次数，并拉近下次复习时间
- 跳过视为一次失败

### FR-305 进度恢复

系统应支持恢复未完成的听写会话。

当前阶段要求：

- 启动应用后自动恢复
- 支持用户手动重置本轮

后续增强要求：

- 进入时允许用户选择“继续”或“重新开始”

## 5.5 语音模块

### FR-401 单词朗读

系统应支持朗读当前单词。

### FR-402 例句朗读

系统应支持朗读当前例句。

### FR-403 语音实现策略

当前版本采用浏览器原生 `speechSynthesis`。

后续可扩展：

- 指定音色
- 指定语速
- 更自然的语音方案
- 离线语音资源包

---

## 5.6 历史与可追踪性

### FR-501 解析历史

系统应记录每次解析事件。

至少包含：

- 时间
- 来源
- 词书
- 模式
- 是否使用 LLM
- 候选数
- 警告信息
- 文本指纹

---

## 5.7 设置模块

### FR-601 One-API 设置

系统应支持用户保存和加载 One-API 设置。

字段包括：

- 启用状态
- Base URL
- Model
- API Key
- Timeout
- System Prompt

要求：

- 前端不回显服务端已保存的 API Key 明文
- 保存后可立即用于增强解析

## 6. 非功能需求

## 6.1 可用性

- 所有长耗时操作都应给出状态提示。
- 关键用户路径应控制在少步骤内完成。
- 页面应在桌面浏览器中正常工作。

## 6.2 稳定性

- LLM 失败不可阻塞主流程。
- OCR、解析、导入失败必须返回可理解错误信息。
- 本地数据库损坏风险应尽量降低。

## 6.3 性能

当前要求：

- 小文本普通解析应具备秒级响应体验。
- One-API 模式在超时后必须自动降级。
- 听写轮次切换应接近即时。

后续目标：

- 建立真实性能基线
- 做并发压测
- 评估大文件解析策略

## 6.4 离线能力

当前版本是本地服务优先设计，具备一定离线特征，但不属于完整离线应用。

后续目标：

- 提升无网络环境下的独立使用能力
- 明确浏览器缓存与资源离线策略

## 6.5 安全性

- API Key 不应在前端明文回显。
- 所有输入都需要做基础校验。
- 服务端应限制请求体大小，避免异常大文件拖垮服务。

## 8. 验收标准

## 8.1 功能验收

- 可以成功导入图片并完成 OCR。
- 可以将文本解析为候选单词。
- 可以将候选词导入词库。
- 可以在词库中搜索、编辑、删除词条。
- 可以开始一轮听写并完成答题。
- 可以正确更新熟练度和下次复习时间。
- One-API 配置后增强模式可以工作。
- One-API 失败时系统可以自动降级。

## 8.2 体验验收

- 页面加载后能清楚知道当前服务状态。
- 上传、解析、导入、听写操作都有明确反馈。
- 用户能在 5 分钟内走完“导入 -> 选词 -> 听写”完整流程。

## 8.3 数据验收

- 重启服务后词库不丢失。
- 重启服务后听写进度可恢复。
- 重复导入同一 lemma 时不应产生明显重复词条。
