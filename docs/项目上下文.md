# HearWords 项目上下文

本文档用于帮助新加入项目的人快速理解：这个项目是什么、现在做到哪了、代码怎么跑、当前重点问题是什么。

---

## 1. 项目是什么

HearWords 是一个围绕“英语单词捕获与内化”设计的本地 Web 应用。

它的核心不是做通用 OCR，也不是做大而全的记忆软件，而是专注于一条学习闭环：

`从图片 / 文档 / 文本里抓取英文词汇 -> 整理成个人词库 -> 通过听写和复习完成内化`

当前代码已经具备可运行的 MVP 闭环。

---

## 2. 当前项目状态

## 2.1 已经跑通的主流程

1. 用户输入文本或上传文件
2. 图片走 OCR，文档走轻量文本提取
3. 服务端解析候选单词
4. 用户筛选后导入词库
5. 用户从词库发起听写
6. 系统根据答题结果更新熟练度和复习时间

## 2.2 当前形态判断

- 这是本地优先的单机版 Web MVP
- 不是正式 SaaS 平台
- 不是多用户系统
- 不是完整移动端 App

它适合作为：

- 继续功能迭代的基础版本
- 功能演示版本
- 单机学习工具原型

---

## 3. 目标用户与产品逻辑

目标用户统一理解为“英语学习者”。

不要把当前产品逻辑设计成复杂多角色系统，当前阶段只需要围绕 Learner 本身展开。

核心产品逻辑是：

- 输入：拍照、截图、导入文件、粘贴文本
- 整理：解析候选词，人工筛选，进入词库
- 内化：听写、复习、记忆状态更新

## 4. 当前技术架构

## 4.1 前端

- 文件：`index.html`、`app.js`、`styles.css`
- 形式：原生浏览器页面
- 状态：单一 `state` 对象管理
- 能力：导入、候选词、词库、历史、听写、设置

## 4.2 后端

- 文件：`server.js`
- 形式：Node 原生 HTTP 服务
- 职责：
  - 静态文件服务
  - API 路由
  - OCR
  - 文本解析
  - LLM 增强解析
  - 听写逻辑

## 4.3 数据层

- 文件：`db.js`
- 形式：SQLite 访问封装
- 核心表：
  - `library_entries`
  - `dictionary_entries`
  - `parse_history`
  - `app_kv`

## 4.4 外部能力

- OCR：`sharp` + `tesseract.js`
- 词典：WordNet 导入 SQLite
- LLM：One-API 兼容 OpenAI Chat Completions
- TTS：浏览器 `speechSynthesis`

## 5. 关键业务对象

## 5.1 Candidate 候选词

解析阶段产生的临时对象。

典型字段：

- `lemma`
- `rawWord`
- `definition`
- `pos`
- `exampleSentence`
- `sourceName`
- `bookName`
- `kept`

## 5.2 Library Entry 词库项

进入正式词库后的持久化对象。

典型字段：

- `id`
- `lemma`
- `rawWord`
- `phonetic`
- `pos`
- `definition`
- `exampleSentence`
- `sourceName`
- `bookName`
- `lastSource`
- `originalForms`
- `masteryLevel`
- `failCount`
- `createdAt`
- `updatedAt`
- `nextReviewTime`

## 5.3 Dictation Session 听写会话

保存一轮听写的状态。

典型字段：

- `queue`
- `index`
- `scope`
- `updatedAt`

## 5.4 Parse History 解析历史

记录每次解析的摘要信息。

典型字段：

- `id`
- `createdAt`
- `sourceName`
- `bookName`
- `mode`
- `llmUsed`
- `warning`
- `fingerprint`
- `candidateCount`

## 6. 关键业务流程

## 6.1 导入与解析

### 现状

- 文本文件：直接读取文本
- 图片文件：发送到后端 OCR
- `pdf/docx/xlsx`：当前通过宽松文本提取方式读取

### 真实边界

图片 OCR 是正式能力。

`pdf/docx/xlsx` 目前只能算“有初步支持”，还不是完整可靠的文档解析方案。这是项目当前非常明确的技术欠债。

## 6.2 普通解析

解析链路：

1. 清洗文本
2. 正则提取 token
3. 过滤数字、噪声、词性标记
4. 做简单词形还原
5. 查询本地词典
6. 形成候选词列表
7. 去重

## 6.3 增强解析

解析链路：

1. 发送文本给 One-API
2. 要求结构化 JSON
3. 读取 `lemma/rawWord/exampleSentence`
4. 用本地词典回填词性和释义
5. 如果失败则回退普通解析

结论：

当前增强模式更像“增强抽词”，不是“完全依赖 LLM 给释义”。

## 6.4 单词导入

候选词导入词库时按 `lemma` 合并。

这意味着：

- 同一个词不会无上限重复新增
- 原始词形会累积到 `originalForms`

## 6.5 听写与复习

系统会根据词库构造一轮队列。

支持两种范围：

- `due`
- `all`

答题结果会更新：

- 熟练度
- 失败次数
- 下次复习时间

## 8. 当前最重要的技术问题

## 8.1 文档解析不是真解析

这是当前最核心问题之一。

现状：

- `pdf/docx/xlsx` 可读到部分文本
- 但没有真正理解这些格式结构

影响：

- 格式兼容性不可控
- 准确率难以达成验收目标

## 8.2 词典质量有限

现状：

- 使用 WordNet
- 释义和词性有基础可用性
- 没有音标

影响：

- 不适合作为考试词书的最终数据源
- 学习体验不够完整

## 8.3 复习算法偏简单

现状：

- 使用固定时间步长

影响：

- 可以跑通 MVP
- 但还不是高质量 SRS

## 8.4 代码集中度较高

现状：

- `server.js` 和 `app.js` 都比较重

影响：

- 增加新功能时容易继续膨胀
- 后续拆模块要尽早规划

## 9. 如何运行项目

```powershell
npm install
```

```powershell
npm run setup:dictionary
```

## 9.3 启动项目

```powershell
npm start
```

```text
http://127.0.0.1:3000
```

```powershell
npm run check
```

## 10. 开发时应默认知道的事情

- 当前项目以本地 MVP 为主，不要过早引入复杂平台设计。
- 增强解析不是主流程兜底，本地普通解析才是基础保障。
- 数据一致性优先级高于炫技式功能。
- `pdf/docx/xlsx` 现有支持不应被误认为“已经正式完成”。
- 听写与复习是这个产品的核心闭环，不要把它弱化成单纯词典展示工具。

---

## 11. 当前推荐开发优先级

### 第一优先级

- 完整 PDF 文本层解析
- DOCX 真实解析
- XLSX 真实解析
- 更稳的文本清洗与词形归一

### 第二优先级

- 更好的词典数据源
- 音标支持
- 更清晰的交互反馈

### 第三优先级

- 学习统计
- 复习算法增强
- 批量操作

### 第四优先级

- 登录
- 云同步
- 多端
