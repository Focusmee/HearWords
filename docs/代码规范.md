# HearWords 代码规范

本文档定义当前项目的代码组织、命名、接口设计、错误处理、数据约束和开发习惯，用于保证后续迭代一致性。

---

## 1. 总体原则

- 先保证功能正确，再追求复杂抽象。
- 保持模块职责单一，避免一个函数同时承担路由、业务、存储、格式转换等多种职责。
- 优先保证代码可读性与可维护性。
- 新增逻辑必须和当前项目结构兼容，不做无收益的大重构。
- 对用户可见的错误，应返回明确、可理解的信息。

---

## 2. 技术栈约束

当前项目技术栈如下：

- 运行时：Node.js
- 服务端：Node 原生 HTTP
- 数据库：SQLite
- OCR：`sharp` + `tesseract.js`
- 前端：原生 HTML / CSS / JavaScript
- 语音：浏览器 `speechSynthesis`

约束要求：

- 在没有明确收益前，不引入重量级框架。
- 如果引入新依赖，必须说明目的、替代方案和影响面。

---

## 3. 文件组织规范

## 3.1 当前结构

- `server.js`：服务端入口与 API
- `db.js`：数据库访问层
- `app.js`：前端交互逻辑
- `index.html`：页面模板
- `styles.css`：样式
- `scripts/`：一次性脚本和初始化脚本
- `data/`：运行数据

## 3.2 后续组织建议

当 `server.js` 或 `app.js` 继续膨胀时，应按职责拆分：

- `server/routes/`
- `server/services/`
- `server/utils/`
- `client/modules/`

拆分原则：

- 路由层只做参数校验和流程调度
- Service 层承载业务逻辑
- DB 层只承载数据访问
- Utils 只放无状态工具函数

---

## 4. 命名规范

## 4.1 通用命名

- 变量、函数：`camelCase`
- 常量：`UPPER_SNAKE_CASE`
- 文件名：优先使用现有风格，当前项目以简单文件名为主
- 数据库字段：`snake_case`
- API 返回 JSON 字段：`camelCase`

## 4.2 命名要求

- 名称必须表达真实语义，不使用 `data1`、`temp2`、`foo`、`bar`
- 布尔值必须体现判断语义，如：
  - `isEnabled`
  - `hasApiKey`
  - `finished`
- 集合名应使用复数，如：
  - `entries`
  - `items`
  - `candidates`

---

## 5. 函数设计规范

- 单个函数尽量只做一件事。
- 超过 50-80 行的函数应考虑拆分。
- 有副作用的函数名称应明确体现动作，如：
  - `saveSession`
  - `writeJson`
  - `upsertLibraryEntry`
- 纯函数应尽量避免读写外部状态。

参数规范：

- 参数对象优先用于多字段输入，避免长参数列表。
- 参数校验尽量靠近入口层完成。

返回值规范：

- 同类函数返回结构应稳定。
- API 层返回 JSON 时保持字段一致性，不随意改名。

---

## 6. 服务端规范

## 6.1 路由层规范

服务端 API 应遵循：

- 先校验请求方法和路径
- 再解析请求体
- 再校验必要参数
- 再调用业务逻辑
- 最后统一返回 JSON

要求：

- 成功响应统一使用 `sendJson`
- 错误响应统一返回 `error` 字段
- 状态码要与错误类型匹配

## 6.2 请求校验

所有 API 入口必须校验：

- 必填字段是否存在
- 字段类型是否正确
- 字符串是否需要 `trim`
- 数值是否需要限制范围

## 6.3 错误处理

必须遵守：

- 不把内部堆栈直接暴露给用户
- 错误文案要能帮助定位问题
- 外部依赖调用失败必须兜底

示例要求：

- OCR 失败：返回明确 OCR 失败信息
- One-API 失败：降级到普通解析，而不是直接中断主流程
- JSON 解析失败：返回明确请求错误

## 6.4 外部依赖调用

调用 LLM、OCR 或文件处理等外部耗时操作时：

- 必须设置超时
- 必须有失败兜底
- 必须避免阻塞整个服务流程

---

## 7. 数据库规范

## 7.1 表设计

- 表字段命名使用 `snake_case`
- 时间字段统一使用时间戳整数
- 需要频繁查询的字段必须建索引

## 7.2 数据访问层约束

- 所有 SQL 集中在 `db.js` 或后续 DB 模块中
- 服务层不直接拼接重复 SQL 逻辑
- 所有数据库写操作必须通过封装函数完成

## 7.3 数据一致性

- 词库导入时，同 lemma 应合并而不是重复新增
- `original_forms` 应保留原始词形信息
- 听写状态更新必须同步写回数据库

## 7.4 JSON 字段处理

- 数据库存 JSON 字段时，写入前必须 `JSON.stringify`
- 读取后必须安全解析
- 解析失败时必须有 fallback

---

## 8. 前端规范

## 8.1 状态管理

当前项目采用单一 `state` 对象。

要求：

- 所有页面状态集中维护
- 修改状态后尽量显式调用渲染函数
- 不要在多个地方偷偷改 DOM 造成状态不一致

## 8.2 DOM 操作

- DOM 引用集中收敛在 `els` 对象
- 尽量避免散落的 `document.getElementById`
- 列表渲染优先使用模板或统一函数

## 8.3 交互反馈

- 所有异步操作都应更新状态文案
- 成功、失败、中性提示需要区分
- 用户关键操作需要明确结果反馈

## 8.4 浏览器能力调用

调用浏览器能力时：

- 先检测能力是否存在
- 降级时不要抛未处理异常

适用场景：

- `speechSynthesis`
- `TextDecoder`
- 文件读取

---

## 9. API 设计规范

## 9.1 基本约定

- 路径使用资源导向
- JSON 返回结构保持稳定
- 不在同一个接口里混入过多无关职责

## 9.2 响应约定

成功响应：

- 可返回 `message`
- 数据字段按语义命名，如：
  - `items`
  - `item`
  - `stats`
  - `session`
  - `current`

失败响应：

- 统一返回：

```json
{
  "error": "错误说明"
}
```

## 9.3 兼容性要求

- 已上线或已被前端依赖的字段不要随意改名
- 如果要调整字段结构，应同步修改前端并补充说明

---

## 10. 解析逻辑规范

## 10.1 普通解析

- 先做文本清洗
- 再做分词
- 再做词形归一
- 再做字典查询
- 最后做候选项去重

## 10.2 增强解析

- LLM 返回只负责抽词，不负责伪造释义
- 词性和释义优先以本地词典为准
- LLM 输出必须按 JSON 结构解析

## 10.3 OCR

- 所有 OCR 图片先预处理
- OCR 中间过程如扩展参数、阈值、角度候选应集中维护
- 不要把 OCR 预处理参数散落多个地方

---

## 11. 文案与注释规范

## 11.1 用户文案

- 直接、明确、可操作
- 避免技术化术语堆叠
- 错误提示尽量告诉用户下一步怎么做

## 11.2 代码注释

- 只注释“为什么这么做”或“这里有风险/边界”
- 不注释显而易见的代码
- 避免过时注释

---

## 12. 测试与校验规范

当前项目至少需要保持：

- `npm run check` 必须通过

后续建议补充：

- 解析函数单元测试
- OCR 关键流程样例测试
- 词库导入去重测试
- 听写流程测试

修改要求：

- 修改核心流程后至少进行一次本地手工验证
- 涉及数据结构变更时必须验证旧数据兼容性

---

## 13. 性能与安全规范

- 请求体大小要有限制
- 文件解析不能无限制吃内存
- LLM 请求必须有 timeout
- 本地配置中的敏感信息不回传明文
- 大文件处理需要明确边界

---

## 14. 提交与变更规范

每次改动应尽量满足：

- 改动目标明确
- 不顺手重写无关模块
- 新增字段要同步文档
- 新增接口要同步前端和说明文档

如果改动涉及：

- 表结构
- API 字段
- 核心学习状态
- 解析链路
