# 产品开发任务清单

这份清单和验收标准（Acceptance Criteria, AC）旨在为开发团队提供一套清晰的“完工定义”。它将产品拆解为核心功能、性能指标和交互体验三个维度。

------

## 一、 产品开发任务清单 (Development Checklist)

### 1. 基础架构与通用模块

- [ ] **多格式导入模块：** 完成对图片（JPG/PNG）、PDF（扫描版/文字版）、Word/Excel 的文件读取。
- [ ] **本地词库构建：** 集成 SQLite，内置 2w+ 核心词汇（含音标、基础释义）。
- [ ] **账户与数据同步：** 实现基于手机号或邮箱的登录，支持词库云端备份。

### 2. 解析引擎模块 (核心)

- [ ] **OCR 流水线：** 实现图像预处理（纠偏、去噪）及文字提取。
- [ ] **逻辑分流器：** 开发“普通解析”与“LLM增强”开关，确保 LLM 模式下 Prompt 能稳定返回 JSON。
- [ ] **单词清洗：** 实现词根还原（Lemmatization），自动过滤标点及数字。

### 3. 单词表与听写模块

- [ ] **交互单词表：** 支持手动勾选、删除、编辑解析出的单词。
- [ ] **TTS 语音系统：** 集成系统原生语音包，实现单词、例句的自动朗读。
- [ ] **听写逻辑：** 开发“拼写检查”逻辑，支持大小写模糊匹配。
- [ ] **复习算法：** 基于艾宾浩斯曲线，实现单词复习任务排期。

------

## 二、 产品验收标准 (Acceptance Criteria)

### 1. 识别与解析验收 (Accuracy)

| **测试项**       | **验收标准**                                                 | **优先级** |
| ---------------- | ------------------------------------------------------------ | ---------- |
| **多格式兼容性** | 相同内容的 PDF、Word、JPG 解析出的单词列表一致率 $>95\%$。   | P0         |
| **普通解析精度** | 本地字典匹配率 $>90\%$（针对标准拼写的考研/雅思核心词汇）。  | P0         |
| **LLM 语义选义** | 在有上下文语境的情况下，LLM 给出的释义准确度显著优于本地字典。 | P1         |
| **解析异常处理** | 当识别到乱码或非单词字符时，系统应自动过滤，不显示在单词表中。 | P1         |

### 2. 听写功能验收 (Interaction)

| **测试项**     | **验收标准**                                                 | **优先级** |
| -------------- | ------------------------------------------------------------ | ---------- |
| **语音实时性** | 点击“播放”按钮，语音响应延迟 $<300ms$。                      | P0         |
| **拼写判分**   | 用户输入完全正确后，自动跳转下一个词；错误时必须显示正确拼写对比。 | P0         |
| **离线可用性** | 断开网络情况下，已导入的单词表仍可正常进行听写和查看。       | P1         |
| **进度保存**   | 听写中途退出，下次进入应能选择从中断处继续或重新开始。       | P1         |

### 3. 技术性能验收 (Performance)

| **测试项**     | **验收标准**                                                 | **优先级** |
| -------------- | ------------------------------------------------------------ | ---------- |
| **冷启动速度** | App 从点击到进入主页的时间 $<2s$。                           | P2         |
| **解析响应**   | 普通模式（10个单词内）解析耗时 $<3s$；LLM模式耗时 $<8s$。    | P1         |
| **并发稳定性** | 支持至少 500 名用户同时在线进行解析请求不崩溃（通过任务队列实现）。 | P1         |
| **容错降级**   | LLM API 超时（默认10秒）后，系统必须弹出“网络波动”并自动切换回普通模式。 | P0         |

------

## 三、 验收流程建议 (Workflow)

1. **Alpha 验收（冒烟测试）：** 程序员演示图片扫描到生成单词表的完整链路。
2. **Beta 验收（边界测试）：** 测试故意上传模糊图片、加密 PDF、或者断网状态下的 App 表现。
3. **用户验收（UX 测试）：** 找几名英语学习者（如备考学生），测试在 5 分钟内能否完成“拍照-选词-听写”的流程。

------

## 四、 给程序员的验收“金律”

> **“如果 OCR 错了，允许修正；如果逻辑崩了，绝对不行。”**

在验收时，我们会重点检查：

- **内存泄漏：** 频繁解析大文件后，App 是否出现卡顿或闪退。
- **数据一致性：** 手机 A 听写完单词，手机 B 登录后，该单词的“熟练度”是否已更新。
- **UI 响应：** 所有耗时操作（上传、解析、同步）必须有 Loading 状态提示。

# 实现逻辑和技术细节

**“去身份化”**的设计非常明智，这会让产品变得更加纯粹、工具化，降低用户的上手心理门槛。无论是为了考研的大学生（如你之前的背景）、辅导孩子的家长，还是单纯想提升词汇量的职场人，他们面对的都是同一个痛点：**如何高效地将纸面/屏幕上的陌生词汇转化为脑子里的长期记忆。**

基于“单一用户角色”的逻辑，我为你重新梳理了产品的实现逻辑和技术细节，确保程序猿们能“闭环”执行。

------

## 一、 产品逻辑重构：以“学习者”为核心

不再区分身份，所有用户统一为 **“Learner”**。功能流简化为：

**输入 (Input) $\rightarrow$ 整理 (Organize) $\rightarrow$ 内化 (Internalize)**

### 1. 核心流程闭环

- **输入：** 拍照、选图、传文件。
- **整理：** 系统自动解析，用户只需点击“保留”或“删除”。
- **内化：** 自动生成的单词表即刻进入听写循环。

------

## 二、 面向开发的：核心模块规格说明书 (Final Tech Spec)

作为 CTO，我要求程序员在实现时必须严格遵循以下**三个标准子系统**：

### 1. 混合解析子系统 (Hybrid Parser)

这是产品的“引擎”。

- **输入协议：** 支持 `.jpg`, `.png`, `.pdf`, `.docx`, `.xlsx`。
- **逻辑分流：**
  - **普通模式（基础）：** `OCR提取` $\rightarrow$ `正则分词` $\rightarrow$ `本地SQLite词库匹配`。
  - **LLM增强模式（高级）：** `OCR提取` $\rightarrow$ `Prompt上下文分析` $\rightarrow$ `JSON解析`。
- **程序员注意：** 必须实现 **“幂等性校验”**。同一个用户重复扫描同一张图，不应产生重复的单词记录。

### 2. 词库与状态管理系统 (Vocabulary State Machine)

单词在系统里不是静态的，而是一个带有状态的“生命体”。

- **字段定义：** * `word_id`, `spelling`, `phonetic`, `definition`, `example_sentence`
  - `mastery_level` (0-5级)
  - `next_review_time` (Unix时间戳)
- **程序员注意：** 必须支持 **“模糊去重”**。比如文档里是 *running*，解析后存入数据库应为词根 *run*，但要保留原句作为例句。

### 3. 听写引擎 (Dictation Engine)

听写不只是拼写检查，而是交互体验。

- **语音驱动：** 优先调用系统自带的 TTS（Text-to-Speech），节省流量；LLM模式下可提供更自然的神经网络发音。
- **反馈逻辑：** * 拼错时：立即高亮错误字母，并记录一次 `fail_count`。
  - 拼对时：播放成功音效，更新该词的遗忘曲线权重。

------

## 三、 关键技术选型 (最终版)

| **组件**       | **推荐方案**                                   | **理由**                                                     |
| -------------- | ---------------------------------------------- | ------------------------------------------------------------ |
| **移动端框架** | **Uni-app** 或 **Flutter**                     | 方便快速构建，且对摄像头调用极其稳定。                       |
| **本地数据库** | **SQLite (Room/Sqlbrite)**                     | 处理万级词库搜索依然流畅，且支持离线听写。                   |
| **OCR 库**     | **Tesseract.js** (前端) / **PaddleOCR** (后端) | 免费且强大，处理清晰文档绰绰有余。                           |
| **LLM 接入**   | **One-API** 聚合网关                           | 一套代码适配所有大模型（如 DeepSeek, GPT, Gemini），方便后续切换成本最低的模型。 |

------

## 四、 程序员避坑指南 (CTO 的最后通牒)

1. **JSON 健壮性：** 强制 LLM 输出 JSON。若 LLM 抽风返回了非标准格式，必须有 `try-catch` 降级逻辑，直接跳回本地查询，不能让界面卡死。
2. **异步加载：** OCR 解析通常需要 2-5 秒。界面必须有丝滑的加载动画（Skeleton Screen），绝对不能阻塞主线程导致 App 无响应。
3. **PDF 解析：** 优先读取 PDF 的文字层（Text Layer），只有当文字层为空（扫描件）时才启动耗能大的 OCR。

这个产品现在定位非常清晰：**一个纯粹的、无干扰的“单词捕获与内化工具”**。

> “你看到的一切英语文本，只要拍张照，3秒钟后它们就会变成你手机里可以听写、可以复习的私人词库。”
