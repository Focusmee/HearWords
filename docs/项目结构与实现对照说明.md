# HearWords 项目结构与实现对照说明

本文档基于当前仓库代码整理，目标是回答 3 个问题：

1. 项目现在有哪些主要模块。
2. 这些模块已经实现了哪些功能，以及大致是如何实现的。
3. 对照 [产品开发任务清单](D:\Projects\HearWords_Web\产品开发任务清单.md)，还有哪些内容没有实现，或只实现了一部分。

---

## 1. 项目结构总览

### 1.1 顶层文件

| 路径 | 作用 |
| --- | --- |
| `server.js` | Node 原生 HTTP 服务，负责静态资源、API、OCR、解析、词库导入、听写流程 |
| `db.js` | SQLite 数据层，负责建表、词库读写、词典查询、解析历史、KV 存储 |
| `app.js` | 前端主逻辑，负责文件导入、调用后端接口、渲染候选词/词库/历史/听写 |
| `index.html` | 页面结构，包含设置、导入、候选词、词库、历史、听写等面板 |
| `styles.css` | 页面样式 |
| `scripts/setup-dictionary.js` | 从 `wordnet-db` 导入本地词典到 SQLite |
| `data/` | 本地数据目录，包含数据库、配置、历史、词库等运行数据 |
| `README.md` | 当前项目说明 |

### 1.2 当前架构形态

这是一个单机版 Web 应用，不是前后端分离部署系统：

- 前端：浏览器打开 `index.html`，由 `app.js` 驱动交互。
- 后端：`server.js` 启动本地 HTTP 服务。
- 数据：SQLite 持久化到 `data/hearwords.db`。
- OCR：后端使用 `sharp + tesseract.js`。
- 词典：本地 WordNet 数据导入 SQLite 后查询。
- TTS：前端直接调用浏览器 `speechSynthesis`。

## 2. 模块说明：实现了什么、如何实现

## 2.1 服务层 `server.js`

### 已实现功能

- 提供健康检查、设置、解析历史、OCR、文本解析、词库管理、听写会话等 API。
- 提供静态文件服务，直接返回 `index.html`、`app.js`、`styles.css`。
- 支持 One-API 配置读取与保存。
- 支持普通解析和增强解析两条链路。
- 支持听写轮次创建、恢复、检查答案、跳过、重置。

### 实现方式

- 基于 Node 原生 `http.createServer`，没有使用 Express。
- 通过 URL 路径手写路由分发。
- 通过 `readJsonBody()` 手动解析请求体 JSON。
- 配置保存在 `data/settings.json`。
- 听写进度保存在 SQLite 的 `app_kv` 表中，键名为 `dictation_session`。

### 关键特点

- 增强模式失败时会自动降级为普通解析。
- OCR 前做了预处理：自动旋转、灰度化、归一化、锐化、阈值化、角度估计纠偏。
- 听写判定使用 `answer.trim().toLowerCase()` 对比 `lemma.toLowerCase()`，支持大小写忽略。

## 2.2 数据层 `db.js`

### 已实现功能

- 自动创建 4 张核心表：
  - `library_entries`
  - `dictionary_entries`
  - `parse_history`
  - `app_kv`
- 提供词库增删改查。
- 提供词典查询与词典数量查询。
- 提供解析历史记录。
- 提供简单 KV 存储。

### 实现方式

- 使用 `sqlite3`。
- 启动时执行 `PRAGMA journal_mode = WAL`，提高并发读写稳定性。
- `library_entries` 中已包含复习字段：
  - `mastery_level`
  - `fail_count`
  - `next_review_time`
- `original_forms` 以 JSON 字符串形式落库，支持保留词形来源。

## 2.3 词典导入脚本 `scripts/setup-dictionary.js`

### 已实现功能

- 能从 `node_modules/wordnet-db/dict` 读取 WordNet 文件。
- 能把词典解析后导入 SQLite。

### 实现方式

- 读取 `index.noun / verb / adj / adv` 和对应 `data.*` 文件。
- 解析出：
  - `lemma`
  - `pos`
  - `definition`
  - `exampleSentence`
  - `rankOrder`
- 导入到 `dictionary_entries` 表。

### 当前边界

- 词典来源是 WordNet 免费词典。
- 已有释义和词性，但没有音标字段来源，因此系统里 `phonetic` 基本为空。

## 2.4 前端页面 `index.html + app.js`

### 已实现功能

- One-API 设置面板。
- 文件导入区。
- 候选词列表。
- 我的单词书。
- 解析历史。
- 听写引擎。

### 实现方式

- `index.html` 负责面板布局。
- `app.js` 负责：
  - 页面启动时并行加载健康状态、设置、词库、会话、历史。
  - 文件读取与 OCR 调用。
  - 解析文本生成候选词。
  - 导入候选词到词库。
  - 渲染筛选后的词库表格。
  - 启动、恢复和操作听写流程。
  - 调用浏览器 `speechSynthesis` 播放单词和例句。

### 交互细节

- 候选词支持勾选保留。
- 词库支持搜索、按复习状态筛选、按词书/来源筛选。
- 词库支持朗读、例句朗读、编辑、删除。
- 听写支持开始待复习、开始全部、检查、跳过、重置本轮。
- 可导出整个词库为 JSON。

## 2.5 导入与解析链路

### 已实现功能

- 文本类文件：`.txt/.md/.csv/.json`
- 二进制文件：`.pdf/.docx/.xlsx`
- 图片文件：`.png/.jpg/.jpeg`

### 实现方式

- 文本文件直接 `file.text()`。
- 图片文件走 `/api/ocr`，由后端做 OCR。
- `pdf/docx/xlsx` 目前不是“真正的格式解析器”，而是前端对二进制内容做宽松文本解码，再抽取可能的英文文本。
- 普通解析链路：
  - 文本清洗
  - 正则提取 token
  - 过滤标点/数字/词性标记
  - 简单词形还原
  - 查询本地词典
  - 生成候选词
- 增强解析链路：
  - 调用 One-API
  - 强制请求 JSON 结果
  - 从返回中抽取 `lemma/rawWord/exampleSentence`
  - 再回查本地词典补齐词性和释义

### 当前边界

- `pdf/docx/xlsx` 只算“轻量读取”，不是完整、稳定的格式解析。
- 没有专门的 PDF 文本层解析器，也没有 DOCX/XLSX 结构化解析库。
- 候选词上限为 80 个。

## 2.6 词库与复习逻辑

### 已实现功能

- 单词导入词库。
- 同 lemma 合并。
- 保存原词形。
- 记录来源、词书、例句、释义、复习状态。
- 按到期时间生成听写队列。

### 实现方式

- 导入时按 `lemma` 合并已有条目。
- 已存在单词时，会合并 `originalForms`，更新定义/词性/来源信息。
- 复习间隔由 `REVIEW_STEPS` 数组控制：
  - 10 分钟
  - 1 小时
  - 8 小时
  - 1 天
  - 3 天
  - 7 天
- 答对：
  - `masteryLevel + 1`
  - 更新 `nextReviewTime`
- 答错或跳过：
  - `failCount + 1`
  - `masteryLevel` 下调或保持低位
  - 下一次复习推到最近的短间隔

## 2.7 听写与语音

### 已实现功能

- 单词朗读。
- 例句朗读。
- 听写检查。
- 错误结果对比显示。
- 听写进度恢复。

### 实现方式

- 语音播放完全依赖浏览器 `SpeechSynthesisUtterance`。
- 听写时展示释义和例句，用户输入拼写后发送到后端校验。
- 拼错时后端返回字符级 diff 文本。
- 启动页面时自动读取未完成会话。

### 当前边界

- 没有自定义音频资源，也没有“答对音效”。
- 没有多发音人、语速包、离线语音资源管理。
- 恢复逻辑是自动恢复，用户可以“重置本轮”，但没有单独的“进入时弹窗选择继续/重新开始”。

## 5. 当前已经形成的产品闭环

从代码看，这个项目已经具备一个可运行 MVP 闭环：

1. 用户导入文本或图片。
2. 系统做 OCR 或文本提取。
3. 系统解析候选单词。
4. 用户筛选后导入词库。
5. 词库支持搜索、编辑、删除、导出。
6. 系统按复习时间生成听写任务。
7. 用户完成听写，系统更新熟练度和下次复习时间。

也就是说，核心“输入 -> 整理 -> 内化”链路已经跑通。

## 6. 还没有实现的关键缺口

如果按产品开发任务清单继续推进，最明显的缺口是：

### P0/P1 级缺口

- 真实的 PDF / DOCX / XLSX 解析器。
- 登录、账户体系、云同步。
- 更可靠的词形还原与去噪清洗。
- 更专业的词库来源，尤其是音标和考试词书。
- 指标化测试：解析精度、格式一致性、性能、并发。

### 体验层缺口

- 更明确的加载态和长任务反馈。
- 更完整的听写反馈体验，比如成功音效、错误高亮 UI。
- 真正的离线应用形态，而不是“依赖本地 Node 服务”的浏览器应用。

### 工程层缺口

- 自动化测试。
- 统一日志与异常监控。
- 并发和大文件处理策略。
- 更清晰的模块拆分，当前逻辑主要集中在 `server.js` 和 `app.js` 两个大文件中。

## 7. 结论

这个项目当前状态不是“还没开始”，而是已经完成了一个可用的本地版 MVP：

- 已完成核心闭环：导入、OCR、解析、词库、听写、复习。
- 已完成基础技术落地：SQLite、WordNet、One-API 降级、浏览器 TTS。
- 未完成的平台化能力：账号、同步、真正多格式解析、性能验收、并发保障。

如果从任务清单角度给一个简化判断：

- 核心功能完成度：约 65% 到 75%
- 产品化完成度：约 40% 到 50%
- 可作为单机 MVP 演示：可以
- 可作为正式产品交付：还不够

---

## 8. 建议下一步

建议优先级如下：

1. 先补真实文档解析能力：PDF 文本层、DOCX、XLSX。
2. 再补词库质量：音标、更多词书、词形还原增强。
3. 然后补账户与云同步。
4. 最后做性能压测和验收指标闭环。
